## Web / Frontend

### Frontend AppSec Checklist (по убыванию severity) с указанием стека и пояснениями для SPA

1. **DOM XSS (dangerous DOM sinks)** *(React, Vue, Vanilla‑JS)*  
   *Пояснение*: внедрение вредоносного кода через небезопасные методы вставки HTML.  
   *Риски*: кража токенов, перехват сессии, фишинг.  
   *Примеры уязвимых мест*:  
   - `dangerouslySetInnerHTML` (React)  
   - `v-html` (Vue)  
   - `innerHTML` / `outerHTML` (Vanilla‑JS)  
   - `document.write`  
   - `eval` / `new Function`

2. **Stored XSS** *(React, Vue, Vanilla‑JS)*  
   *Пояснение*: сохранение вредоносного кода на сервере, который отображается всем пользователям.  
   *Риски*: массовое заражение пользователей, долгосрочное воздействие.  
   *Примеры*:  
   - Несанитизированные комментарии/профили  
   - Markdown без очистки (например, через `marked` без `sanitize: true`)

3. **Reflected XSS через URL** *(React Router, Vue Router, Vanilla‑JS)*  
   *Пояснение*: исполнение кода из параметров URL без валидации.  
   *Риски*: фишинговые атаки, распространение через ссылки.  
   *Примеры*:  
   - Вставка `location.search` / `location.hash` в DOM  
   - Небезопасная обработка query‑параметров

4. **Утечка токенов/секретов** *(React, Vue, Vanilla‑JS, Vite)*  
   *Пояснение*: раскрытие аутентификационных данных через клиентские механизмы.  
   *Риски*: несанкционированный доступ к API, кража учётных данных.  
   *Примеры*:  
   - Хранение JWT в `localStorage`/`sessionStorage`  
   - Передача токенов в query‑строке  
   - Жёстко закодированные `VITE_API_KEY` и аналогичные

5. **Prototype Pollution** *(React, Vue, Vanilla‑JS)*  
   *Пояснение*: модификация прототипов объектов для изменения логики приложения.  
   *Риски*: XSS, обход валидации, повреждение состояния.  
   *Примеры*:  
   - Небезопасный `lodash.merge` с внешними данными  
   - `Object.assign` с недоверенными источниками  
   - Использование `__proto__`

6. **Обход бизнес‑логики на клиенте** *(React, Vue)*  
   *Пояснение*: проверка прав доступа только на фронтенде без серверной валидации.  
   *Риски*: выполнение неавторизованных действий через прямые API‑запросы.  
   *Примеры*:  
   - Скрытие кнопок вместо блокировки API  
   - Проверка ролей только в UI  
   - Использование `disabled` полей как «защиты»

7. **Open Redirect** *(React Router, Vue Router, Vanilla‑JS)*  
   *Пояснение*: перенаправление пользователя на произвольный URL.  
   *Риски*: фишинг, переход на вредоносные сайты.  
   *Примеры*:  
   - `window.location = userInput`  
   - Параметр `?redirect=...` без валидации

8. **CSRF в SPA** *(React, Vue, Vite)*  
   *Пояснение*: принудительное выполнение действий от имени пользователя.  
   *Риски*: несанкционированные изменения данных.  
   *Примеры*:  
   - Отсутствие CSRF‑токенов для POST‑запросов  
   - Неправильная настройка `SameSite` для куки

9. **Clickjacking** *(Nginx, Vite build)*  
   *Пояснение*: скрытое наложение интерфейса для перехвата кликов.  
   *Риски*: обман пользователя при выполнении действий.  
   *Примеры*:  
   - Отсутствие `X-Frame-Options`  
   - Нет CSP `frame-ancestors`

10. **Утечка source maps** *(Vite, Webpack)*  
    *Пояснение*: раскрытие исходного кода через map‑файлы.  
    *Риски*: упрощение реверс‑инжиниринга, поиск уязвимостей.  
    *Примеры*:  
    - `build.sourcemap = true` в продакшене  
    - Публикация `*.map` файлов

11. **Небезопасные сторонние скрипты** *(React, Vue, Vite)*  
    *Пояснение*: загрузка кода из ненадёжных источников.  
    *Риски*: внедрение вредоносного JS через зависимости.  
    *Примеры*:  
    - CDN без `integrity` (SRI)  
    - Динамическое создание `<script>`

12. **Unsafe `postMessage`** *(React, Vue, Vanilla‑JS)*  
    *Пояснение*: междоменная коммуникация без проверки источника.  
    *Риски*: перехват данных, XSS.  
    *Примеры*:  
    - Отсутствие проверки `event.origin`  
    - Обработка `event.data` без валидации

13. **Утечка чувствительных данных в bundle** *(Vite, React, Vue)*  
    *Пояснение*: хранение секретов в клиентском коде.  
    *Риски*: раскрытие API‑ключей, внутренней логики.  
    *Примеры*:  
    - Жёстко закодированные ключи  
    - Debug‑флаги в продакшене

14. **Abuse Web Storage** *(React, Vue, Vanilla‑JS)*  
    *Пояснение*: небезопасное хранение данных в клиентских хранилищах.  
    *Риски*: кража PII через XSS.  
    *Примеры*:  
    - Хранение персональных данных в `localStorage`  
    - Refresh‑токены без защиты

15. **Resource Exhaustion** *(React, Vue)*  
    *Пояснение*: перегрузка клиента вычислительно тяжёлыми операциями.  
    *Риски*: зависание вкладки, DoS.  
    *Примеры*:  
    - Рендер больших массивов без виртуализации  
    - Тяжёлые regex в обработчиках

16. **Local File Exposure** *(React, Vue)*  
    *Пояснение*: несанкционированный доступ к локальным файлам пользователя.  
    *Риски*: утечка конфиденциальных данных.  
    *Примеры*:  
    - `FileReader` без проверки MIME‑типа  
    - Вставка локальных файлов в DOM через `URL.createObjectURL`


17. **CORS Misconfiguration** *(React, Vue, архитектура SPA)*  
    *Пояснение*: неправильное понимание CORS как механизма безопасности.  
    *Риски*: ошибочная уверенность в защите.  
    *Примеры*:  
    - Использование `*` с `credentials: true`  
    - Полагание на CORS для блокировки запросов

18. **Dependency Risks** *(React, Vue, Vite, npm/yarn)*  
    *Пояснение*: уязвимости в сторонних библиотеках.  
    *Риски*: удалённое выполнение кода через зависимости.  
    *Примеры*:  
    - Нефиксированные версии (`^`, `~`)  
    - Уязвимые пакеты (проверяется через `npm audit`)

19. **Dev Proxy Misconfiguration** *(Vite, Webpack Dev Server)*  
    *Пояснение*: доступ к внутренним сервисам через фронтенд‑прокси.  
    *Риски*: компрометация внутренних API.  
    *Примеры*:  
    - Проксирование внутренних endpoints  
    - Отсутствие фильтрации `target`

20. **Hardcoded Secrets в коде** *(React, Vue, Vite)*  
    *Пояснение*: секреты, попавшие в исходники или конфиги.  
    *Риски*: лёгкое обнаружение через анализ кода.  
    *Примеры*:  
    - Ключи API в `.env`  
    - `.env` в публичном репозитории

**Примечания**:  
- **Severity** может варьироваться в зависимости от контекста приложения.  
- Список фокусируется **исключительно на фронтендовых уязвимостях** (без серверных рисков).  
- Для аудита рекомендуется комбинировать автоматизированные инструменты: SAST (Semgrep, ESLint), SCA (Trivy, npm audit), DAST (OWASP ZAP) с ручным тестированием.

### Общий фронтенд‑AppSec‑чек‑лист (20 пунктов)  

*(универсальный, для любых веб‑приложений: SPA, SSR, статические сайты)*

1. **DOM XSS (dangerous DOM sinks)**  
   *Пояснение*: внедрение вредоносного кода через небезопасные методы манипуляции DOM.  
   *Риски*: кража токенов, перехват сессии, фишинг.  
   *Примеры уязвимых мест*:  
   - `innerHTML` / `outerHTML`  
   - `document.write()`  
   - `eval()` / `new Function()`  
   - `setTimeout()`/`setInterval()` с строковыми аргументами  
   - Небезопасная работа с `location.hash`, `location.search`  
   *Рекомендации*: использовать безопасные альтернативы (например, `textContent`), санитировать ввод.

2. **Stored XSS (пользовательский контент)**  
   *Пояснение*: сохранение вредоносного кода на сервере, который отображается всем пользователям.  
   *Риски*: массовое заражение, долгосрочное воздействие.  
   *Примеры*:  
   - Несанитизированные комментарии, профили, описания  
   - Markdown/HTML‑редакторы без очистки  
   *Рекомендации*: применять библиотеки санитизации (например, DOMPurify), ограничивать HTML‑теги.

3. **Reflected XSS через URL**  
   *Пояснение*: исполнение кода из параметров URL без валидации.  
   *Риски*: фишинговые атаки, распространение через ссылки.  
   *Примеры*:  
   - Вставка `location.search` в DOM  
   - Динамическое формирование URL на основе пользовательского ввода  
   *Рекомендации*: проверять и кодировать параметры URL перед использованием.

4. **Утечка чувствительных данных**  
   *Пояснение*: раскрытие PII, токенов или ключей через клиентские механизмы.  
   *Риски*: несанкционированный доступ, кража данных.  
   *Примеры*:  
   - Хранение JWT в `localStorage`  
   - Передача токенов в query‑строке  
   - Логирование PII в консоли  
   *Рекомендации*: использовать `HttpOnly` + `Secure` cookies для токенов, избегать хранения секретов в коде.

5. **Prototype Pollution**  
   *Пояснение*: модификация прототипов объектов для изменения логики приложения.  
   *Риски*: XSS, обход валидации, повреждение состояния.  
   *Примеры*:  
   - Небезопасный `lodash.merge` с внешними данными  
   - `Object.assign` с недоверенными источниками  
   - Использование `__proto__`  
   *Рекомендации*: избегать слияния объектов с недоверенными данными, использовать строгие схемы валидации.

6. **Небезопасные сторонние скрипты**  
   *Пояснение*: загрузка кода из ненадёжных источников.  
   *Риски*: внедрение вредоносного JS через зависимости.  
   *Примеры*:  
   - CDN без `integrity` (Subresource Integrity, SRI)  
   - Динамическое создание `<script>`  
   *Рекомендации*: всегда указывать `integrity` для CDN‑скриптов, ограничивать источники через CSP.

7. **CSP bypass (слабая Content Security Policy)**  
   *Пояснение*: недостаточная защита от загрузки вредоносных ресурсов.  
   *Риски*: выполнение произвольного кода, кража данных.  
   *Примеры*:  
   - Отсутствие `script-src` или использование `*`  
   - Разрешённые `inline-scripts`/`eval`  
   *Рекомендации*:  
   - Использовать `nonce` или `hash` для разрешённых скриптов  
   - Запретить `unsafe-inline`, `unsafe-eval`  
   - Указать строгие `default-src`, `object-src`

8. **Clickjacking**  
   *Пояснение*: скрытое наложение интерфейса для перехвата кликов.  
   *Риски*: обман пользователя при выполнении действий.  
   *Примеры*:  
   - Отсутствие `X-Frame-Options`  
   - Нет CSP `frame-ancestors`  
   *Рекомендации*:  
   - Установить `X-Frame-Options: DENY` или `SAMEORIGIN`  
   - Добавить CSP: `frame-ancestors 'self'`

9. **Source map leakage**  
   *Пояснение*: раскрытие исходного кода через map‑файлы.  
   *Риски*: упрощение реверс‑инжиниринга, поиск уязвимостей.  
   *Примеры*:  
   - `build.sourcemap = true` в продакшене  
   - Публикация `*.map` файлов  
   *Рекомендации*: отключать source maps в продакшене, блокировать доступ к `*.map` через серверную конфигурацию.

10. **Dependency risks (уязвимые пакеты)**  
    *Пояснение*: компрометация приложения через зависимости.  
    *Риски*: удалённое выполнение кода, кража данных.  
    *Примеры*:  
    - Уязвимые версии библиотек (проверяется через `npm audit`)  
    - Dependency confusion (публикация пакетов с именами внутренних модулей)  
    *Рекомендации*:  
    - Фиксировать версии зависимостей (`1.2.3`, а не `^1.2.0`)  
    - Использовать scopes в npm для внутренних пакетов  
    - Регулярно проверять зависимости на уязвимости

11. **Hardcoded secrets (жёстко закодированные секреты)**  
    *Пояснение*: секреты, попавшие в исходники или конфиги.  
    *Риски*: лёгкое обнаружение через анализ кода.  
    *Примеры*:  
    - Ключи API в `.env`  
    - `.env` в публичном репозитории  
    *Рекомендации*:  
    - Использовать переменные окружения в CI/CD  
    - Сканировать код на наличие секретов (например, через `git-secrets`)

12. **Некорректная обработка MIME‑типов (MIME‑sniffing)**  
    *Пояснение*: браузер угадывает тип файла, игнорируя `Content-Type`.  
    *Риски*: исполнение вредоносного кода под видом «безопасного» файла.  
    *Примеры*:  
    - Загрузка `.txt` с JS‑кодом  
    - Отсутствие `X-Content-Type-Options: nosniff`  
    *Рекомендации*:  
    - Устанавливать `X-Content-Type-Options: nosniff`  
    - Проверять MIME‑тип на сервере перед загрузкой

13. **Отсутствие HSTS (HTTP Strict Transport Security)**  
    *Пояснение*: риск downgrade‑атак (перехват HTTP‑трафика).  
    *Риски*: перехват куки, токенов, PII.  
    *Примеры*:  
    - Сайт доступен по HTTP  
    - Нет заголовка `Strict-Transport-Security`  
    *Рекомендации*:  
    - Включить HSTS с `max-age=31536000; includeSubDomains; preload`  
    - Добавить домен в HSTS Preload List

14. **Слабая защита форм**  
    *Пояснение*: отсутствие механизмов против автоматизации.  
    *Риски*: спам, брутфорс, SQLi через формы.  
    *Примеры*:  
    - Формы без CAPTCHA  
    - Отсутствие rate‑limiting для отправки  
    *Рекомендации*:  
    - Добавлять CAPTCHA (например, Google reCAPTCHA)  
    - Ограничивать количество запросов с одного IP

15. **Утечка информации в заголовках**  
    *Пояснение*: раскрытие деталей сервера/приложения через HTTP‑заголовки.  
    *Риски*: помощь злоумышленнику в подборе атак.  
    *Примеры*:  
    - `X-Powered-By: PHP/7.4`  
    - `Server: nginx/1.18`  
    *Рекомендации*:  
    - Удалять лишние заголовки на сервере  
    - Использовать общие значения (например, `Server: webserver`)

16. **Open Redirect**  
    *Пояснение*: перенаправление пользователя на произвольный URL.  
    *Риски*: фишинг, переход на вредоносные сайты.  
    *Примеры*:  
    - `window.location = userInput`  
    - Параметр `?redirect=...` без валидации  
    *Рекомендации*:  
    - Валидировать целевые URL (только разрешённые домены)  
    - Использовать белый список разрешённых redirect‑путей

17. **Resource Exhaustion (клиентский DoS)**  
    *Пояснение*: перегрузка клиента вычислительно тяжёлыми операциями.  
    *Риски*: зависание вкладки, отказ в обслуживании.  
    *Примеры*:  
    - Рендер больших массивов без виртуализации  
    - Тяжёлые regex в обработчиках  
    *Рекомендации*:  
    - Использовать виртуализацию списков (например, `react-window`)  
    - Оптимизировать регулярные выражения

18. **Local File Exposure**  
   *Пояснение*: несанкционированный доступ к локальным файлам пользователя через клиентский код.  
   *Риски*: утечка конфиденциальных данных (документов, фото, токенов).  
   *Примеры уязвимых мест*:  
   - `FileReader` без проверки MIME‑типа (например, чтение `.txt`, `.json`).  
   - Использование `URL.createObjectURL()` для отображения локальных файлов в DOM.  
   - Загрузка файлов без валидации расширения.  
   *Рекомендации*:  
   - Проверять MIME‑тип и расширение файлов перед обработкой.  
   - Ограничивать типы загружаемых файлов (белый список).  
   - Не вставлять локальные файлы напрямую в DOM без санитизации.

19. **Небезопасные HTTP‑методы**  
   *Пояснение*: уязвимость из‑за разрешённых опасных HTTP‑методов (OPTIONS, TRACE, PUT, DELETE).  
   *Риски*:  
   - TRACE — возможность перехвата куки через XST (Cross‑Site Tracing).  
   - OPTIONS — раскрытие информации о разрешённых методах сервера.  
   - PUT/DELETE — несанкционированное изменение данных.  
   *Примеры*:  
   - Сервер отвечает на `TRACE / HTTP/1.1`.  
   - Разрешены `PUT`/`DELETE` для неавторизованных пользователей.  
   *Рекомендации*:  
   - Отключить TRACE и OPTIONS на уровне сервера (Nginx/Apache).  
   - Использовать CORS для ограничения методов (`Access-Control-Allow-Methods`).  
   - Для REST‑API — аутентификация и авторизация для всех методов.

20. **Отсутствие `Content-Type: nosniff`**  
   *Пояснение*: браузер может игнорировать заголовок `Content-Type` и «угадывать» тип файла (MIME‑sniffing).  
   *Риски*:  
   - Исполнение JS‑кода из файла с типом `text/plain`.  
   - Загрузка вредоносного CSS/JS под видом изображений.  
   *Примеры*:  
   - Файл `.jpg` содержит JS‑код, но браузер его исполняет.  
   - Отсутствие заголовка `X-Content-Type-Options: nosniff`.  
   *Рекомендации*:  
   - Всегда добавлять заголовок `X-Content-Type-Options: nosniff` на сервере.  
   - Явно указывать `Content-Type` для всех ресурсов (например, `text/javascript` для `.js`).  
   - Настраивать CSP для блокировки подозрительных типов (например, `script-src 'self'`).

**Итоговые рекомендации по использованию чек‑листа:**

1. **Для аудита**:  
   - Пройтись по каждому пункту, проверяя наличие уязвимостей в коде и конфигурации.  
   - Использовать автоматизированные инструменты (ESLint, OWASP ZAP, Lighthouse, `npm audit`).  

2. **Для разработки**:  
   - Внедрить пункты чек‑листа в процесс CI/CD (например, сканирование зависимостей, проверка заголовков).  
   - Обучить команду базовым принципам безопасности (CSP, санитизация, хранение токенов).

3. **Для эксплуатации**:  
   - Регулярно обновлять зависимости и серверные настройки.  
   - Мониторить логи на признаки атак (например, подозрительные запросы к `/trace`).

4. **Для разных типов приложений**:  
   - **SPA**: уделить внимание пунктам 1–6, 10–11 (XSS, токены, зависимости).  
   - **Статические сайты**: сосредоточиться на 7–9, 12–15, 17–20 (CSP, заголовки, MIME).  
   - **SSR‑приложения**: проверить все пункты, особенно серверные заголовки (HSTS, CSP, X‑Frame‑Options).

5. **Для углублённого анализа**:  
   - По каждому пункту можно провести пентест (например, попытка XSS через `innerHTML`, проверка CSP bypass).  
   - Использовать OWASP Top 10 как дополнение к чек‑листу.

### Итоговый AppSec Frontend Checklist (production‑ready)

Чек‑лист охватывает **критические и высокоприоритетные** уязвимости фронтенда. Реализация пунктов закрывает ~90–95 % реальных рисков.

---

#### 1. XSS‑защита (Critical)

**Цель**: предотвратить внедрение и исполнение произвольного кода.  

- **Авто‑экранирование**  
  - Использовать фреймворки с встроенным санизированием (React, Vue).  
  - Не отключать защитные механизмы (например, React’s JSX‑экранирование).  

- **Запрет опасных методов**  
  - Отказаться от:  
    - `dangerouslySetInnerHTML` (React);  
    - `v-html` (Vue);  
    - `innerHTML` / `outerHTML` (Vanilla JS).  

- **Санитизация HTML**  
  - Если рендеринг HTML необходим — использовать **DOMPurify** с настройками по умолчанию.  
  - Ограничить разрешённые теги и атрибуты.  

- **Запрет динамического кода**  
  - Запретить:  
    - `eval()`;  
    - `new Function()`;  
    - Динамическую загрузку скриптов (`createElement('script')`).  

- **Строгий CSP (Content Security Policy)**  
  ```http
  Content-Security-Policy:
    default-src 'self';
    script-src 'self' 'unsafe-hash' 'nonce-<value>';
    object-src 'none';
    base-uri 'none';
    frame-ancestors 'none';
    require-trusted-types-for 'script';
  ```
  - **Важно**:  
    - Запретить `'unsafe-inline'`;  
    - Использовать `nonce` или `hash` для разрешённых скриптов;  
    - Включить **Trusted Types** (если поддерживается браузером).  

---

#### 2. Token Security (Critical)

**Цель**: защитить аутентификационные данные от кражи и компрометации.  

- **Хранение access token**  
  - Только в `HttpOnly` + `Secure` + `SameSite=Strict` cookie.  
  - Срок жизни (TTL): короткий (например, 15–30 мин).  

- **Refresh token**  
  - Хранить в `HttpOnly` cookie с `Secure` и `SameSite`.  
  - Реализовать ротацию (каждый refresh‑запрос выдаёт новый токен).  

- **Запрещённые места хранения**  
  - `localStorage`;  
  - `sessionStorage`;  
  - Query‑строка URL.  

- **Silent refresh**  
  - Выполнять через backend‑endpoint (не клиентский JS).  

- **CSRF‑защита при cookie‑аутентификации**  
  - Установить `SameSite=Strict/Lax` для куки.  
  - Использовать CSRF‑токены для state‑changing запросов (POST/PUT/DELETE).  
  - Проверять заголовок `Origin` на сервере.  

---

#### 3. Клиент не является trust boundary (Critical)

**Цель**: исключить доверие к клиентским проверкам.  

- **Авторизация на backend**  
  - Все проверки прав доступа — только на сервере.  
  - Клиент не должен принимать решения о доступе (например, «скрыть кнопку»).  

- **Проверка ролей**  
  - Роли и разрешения проверяются исключительно на сервере.  
  - Клиент получает только UI‑инструкции (например, «показать/скрыть»), но не логику.  

- **Защита от подмены ID**  
  - Любые ID (пользователя, ресурса) проверяются на сервере.  
  - Не передавать чувствительные ID в клиентских данных без валидации.  
- **UI‑элементы**  
  - Не полагаться на `hidden` / `disabled` для защиты (это лишь визуальная блокировка).  

---

#### 4. Supply Chain / Dependency Security (High)

**Цель**: минимизировать риски компрометации через зависимости.  

- **Фиксированные версии**  
  - Указывать точные версии в `package.json` (без `^`, `~`).  
  - Использовать lock‑файл (`package-lock.json`, `yarn.lock`).  

- **Сканирование зависимостей**  
  - Регулярно проверять уязвимости через:  
    - `npm audit`;  
    - SCA‑инструменты (Snyk, Dependabot).  

- **Проверка издателей**  
  - Убедиться, что пакеты опубликованы доверенными авторами.  
  - Избегать малоизвестных или форкнутых пакетов.  

- **Минимизация зависимостей**  
  - Удалять неиспользуемые пакеты.  
  - Предпочитать встроенные решения (например, `fetch` вместо `axios`, если достаточно).  

- **Subresource Integrity (SRI)**  
  - Для CDN‑скриптов указывать `integrity` и `crossorigin`.  

---

#### 5. Build Security (High)

**Цель**: обезопасить процесс сборки и финальный артефакт.  

- **Секретные переменные**  
  - Не хранить секреты в `.env` или VITE_*‑переменных.  
  - Использовать CI/CD‑секреты.  

- **Production‑сборка**  
  - Отключить source maps (`build.sourcemap = false`).  
  - Установить `debug = false`.  

- **Проверка бандла**  
  - Сканировать финальный bundle на наличие секретов (например, через `grep` или специализированные инструменты).  

- **Подпись артефактов**  
  - В enterprise‑среде: подписывать сборки (например, через GPG).  

---

#### 6. Storage & Browser APIs (High)

**Цель**: безопасно использовать клиентские хранилища и API.  


- **localStorage / sessionStorage**  
  - Не хранить PII (персональные данные), токены, платёжную информацию.  
  - Использовать только для нечувствительных данных (настройки UI).  

- **postMessage**  
  - Всегда проверять `event.origin` перед обработкой сообщения.  
  - Валидировать `event.data`.  
- **iframe‑безопасность**  
  - Ограничить источники через CSP: `frame-ancestors 'self'`.  
- **Заголовки безопасности**  
  - Установить `X-Content-Type-Options: nosniff` (защита от MIME‑sniffing).  

---

#### 7. Redirect & Navigation (Medium)

**Цель**: предотвратить фишинг и open redirect.  

- **Whitelist для redirect**  
  - Разрешать перенаправления только на домены из белого списка.  
  - Пример:  
    ```js
    const allowedDomains = ['example.com', 'trusted.org'];
    if (allowedDomains.includes(new URL(url).hostname)) {
      window.location.href = url;
    }
    ```
- **Валидация URL**  
  - Использовать `new URL()` для проверки структуры URL.  
  - Не вставлять `userInput` напрямую в `location.href`.  

---

#### 8. Runtime Hardening (Medium)

**Цель**: усилить защиту на уровне HTTP‑заголовков и runtime‑поведения.  


- **HSTS (HTTP Strict Transport Security)**  
  - Включить с параметрами:  
    ```http
    Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
    ```

- **Referrer‑Policy**  
  - Установить:  
    ```http
    Referrer-Policy: strict-origin
    ```  
  - Ограничивает передачу Referer‑заголовка.  

- **Permissions‑Policy**  
  - Контролировать доступ к API (камера, микрофон, геолокация):  
    ```http
    Permissions-Policy: camera=(), microphone=(), geolocation=()
    ```

- **Запрет inline‑JS**  
  - Через CSP (`'unsafe-inline'` запрещён).  
  - Все скрипты — только через внешние файлы с `nonce`/`hash`.  

- **Service Worker**  
  - Не кэшировать чувствительные API‑эндпоинты.  
  - Ограничить scope (например, `/sw.js` вместо `/`).  

---

### Итог

**Если реализовано всё выше:**  
- Закрывается ~90–95 % критических фронтенд‑уязвимостей.  
- Основные оставшиеся риски:  
  - 0‑day в библиотеках/фреймворках;  
  - Логические ошибки в бизнес‑логике;  
  - Уязвимости на стороне backend.  

**Главное:**

Frontend безопасность — это:

- Минимизировать доверие к JS.
- Минимизировать доступ к токенам.
- Запретить выполнение произвольного кода.
- Перенести контроль на backend.
